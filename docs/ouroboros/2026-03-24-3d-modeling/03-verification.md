# SIKMUBYNCH - 3D 모델링 파이프라인 검증

> Ouroboros Phase 3 | 2026-03-24 | 모호성: 16%

## 검증 전략

**Godot 실행 직접 확인 + 간단 스모크 테스트**

```
모델 교체 완료 프로세스:
  1. 모든 GLB 모델을 project/assets/models/에 배치
  2. 코드 수정 (_build_mesh() 함수들)
  3. Godot 실행 → 직접 플레이
  4. 스모크 테스트 (3웨이브 플레이, 크래시/버그 없으면 OK)
  5. 실루엣 구분 + 스타일 통일성 확인
```

---

## E2E 시나리오

### S-MODEL-1: 전체 엔티티 렌더링 확인

**전제 조건**: 모든 GLB 파일이 `project/assets/models/`에 배치됨
**스텝**:
1. Godot 에디터에서 프로젝트 열기
2. 게임 실행 (F5)
3. 타이틀 화면 → 게임 시작
4. 건물 배치: HQ(자동), Tower, Barracks, Barricade, Miner, Buff Tower 각 1개
5. 1~3 웨이브 진행하며 적 6종 + 유닛 4종 출현 확인
6. 발사체 + 미네랄 오브 확인

**기대 결과**:
- 모든 엔티티가 GLB 모델로 렌더링됨 (프리미티브 아님)
- 그림자가 모델 아래에 정상 표시
- HP바가 모델 위에 정상 표시

**성공 판정**: 20개 엔티티 중 18개 이상이 GLB로 렌더링되면 통과

### S-MODEL-2: 실루엣 구분 테스트

**전제 조건**: S-MODEL-1 통과
**스텝**:
1. 게임 실행 → 3웨이브 이상 진행
2. 줌 아웃 (마우스 휠) → 전체 맵 보기
3. 적 6종이 동시에 화면에 있을 때 실루엣으로 구분 가능한지 확인
4. 유닛 4종이 동시에 화면에 있을 때 실루엣으로 구분 가능한지 확인
5. 건물 5종이 나란히 있을 때 구분 가능한지 확인

**기대 결과**:
- 줌 아웃 상태에서도 각 엔티티 타입이 형태만으로 구분됨
- 색상 없이 형태만으로도 "이건 러셔, 이건 탱크" 판별 가능

**성공 판정**: 모든 엔티티가 실루엣으로 구분 가능하면 통과

### S-MODEL-3: 스타일 통일성 테스트

**전제 조건**: S-MODEL-1 통과
**스텝**:
1. 게임 실행 → 모든 엔티티 타입이 화면에 나타나도록 플레이
2. 전체적으로 "같은 세계관" 느낌인지 주관적 판단
3. 특별히 튀는 모델이 있는지 확인

**기대 결과**:
- 다크 판타지 + 밀리터리 분위기 일관
- 로우폴리 수준이 비슷 (하나만 하이폴리이거나 하나만 너무 단순하지 않음)

**성공 판정**: "같은 게임에서 나온 것 같다"는 느낌이면 통과. 1~2개 약간 이질적이면 해당 모델만 재생성.

### S-MODEL-4: 기존 기능 스모크 테스트

**전제 조건**: S-MODEL-1 통과
**스텝**:
1. 게임 시작 → 3웨이브 플레이
2. 확인 항목:
   - [ ] 건물 배치 (클릭) 정상
   - [ ] 건물 레벨업 (클릭) → 스케일 증가 보임
   - [ ] 건물 철거 (우클릭) 정상
   - [ ] 타워 자동 사격 + 발사체 표시
   - [ ] 적 이동 + 건물 충돌/공격
   - [ ] 유닛 생산 + 자동 전투
   - [ ] 적 사망 → 미네랄 오브 비행
   - [ ] 피격 플래시 (색상 변화) 동작
   - [ ] 시너지 바 표시
   - [ ] 보상 카드 3택 표시
3. 크래시/프리즈 없이 3웨이브 완료

**기대 결과**: 기존 기능 전부 정상 동작

**성공 판정**: 체크리스트 10개 중 9개 이상 통과 + 크래시 없음

### S-MODEL-5: 성능 테스트

**전제 조건**: S-MODEL-4 통과
**스텝**:
1. 게임 실행 → F3 디버그 오버레이 켜기
2. 5웨이브 이상 진행하여 200+ 적 동시 출현
3. FPS 수치 확인

**기대 결과**:
- 200적 동시: 60fps 이상
- 500적 동시: 45fps 이상 (프리미티브보다 저하 가능하지만 플레이 가능 수준)

**성공 판정**: 200적 60fps 유지. 500적에서 30fps 미만이면 해당 모델 폴리곤 축소 필요.

## 엣지케이스 시나리오

### E-MODEL-1: GLB 로드 실패

**상황**: 특정 GLB 파일이 손상되었거나 경로가 틀린 경우
**기대 동작**: 폴백으로 기존 프리미티브 메시 표시 (코드에 폴백 로직 포함)
**검증**: 의도적으로 GLB 파일 하나를 삭제 후 게임 실행 → 해당 엔티티만 프리미티브로 표시

### E-MODEL-2: 스케일 불일치

**상황**: GLB 모델의 크기가 게임 좌표계와 맞지 않는 경우
**기대 동작**: 모델이 너무 크거나 작음 → Godot 임포트 설정 또는 코드에서 scale 조정
**검증**: 각 모델의 크기를 기존 프리미티브와 비교 (그리드 셀 기준)

### E-MODEL-3: Origin Point 불일치

**상황**: GLB 모델의 원점이 바닥 중앙이 아닌 경우
**기대 동작**: 모델이 공중에 뜨거나 바닥에 묻힘 → position.y 조정
**검증**: 모든 엔티티가 지면에 올바르게 접지되어 있는지 시각적 확인

### E-MODEL-4: 높은 폴리곤 (Blender 없이)

**상황**: Hunyuan3D 출력이 ~1,000 tri보다 훨씬 높은 경우 (10K+)
**기대 동작**:
1. Hunyuan3D face_reducer 파라미터로 재생성
2. 그래도 높으면 Godot 임포트 시 메시 단순화 옵션 사용
3. 최후 수단: 높은 폴리곤 그대로 사용하되 성능 모니터링
**검증**: F3 디버그 오버레이로 총 삼각형 수 확인

### E-MODEL-5: Material Override 충돌

**상황**: GLB 내장 material과 코드 material_override가 충돌
**기대 동작**: material_override가 항상 우선 (Godot 기본 동작)
**검증**: 피격 플래시, 레벨업 색상 변화가 정상 동작하는지 확인

## 모델 품질 실패 시 대응

AI 재생성 (최대 5회):
1. 프롬프트 수정 (더 구체적인 설명 추가)
2. 다른 시드/변형으로 재생성
3. Shape only → Shape+Texture 모드 변경 시도
4. 5회 실패 시 → 해당 엔티티는 프리미티브 유지, 나중에 재시도

## 성공/실패 판정 매트릭스

| 카테고리 | 전체 통과 | 부분 통과 | 실패 |
|---------|-----------|-----------|------|
| 렌더링 | 20/20 GLB 렌더링 | 18+/20 GLB | 17 이하 |
| 실루엣 구분 | 모든 엔티티 구분 가능 | 80%+ 구분 | 구분 불가 |
| 스타일 통일 | 전체 통일 | 1~2개 이질적 | 3개+ 이질적 |

**전체 통과 = 출시 가능 비주얼**
**부분 통과 = 이질적 모델만 재생성 후 재검증**
**실패 = 파이프라인 재검토 (도구 변경 또는 Blender 도입 고려)**

## 검증 실행 순서

```
1. S-MODEL-1 (전체 렌더링)     ← 반드시 첫 번째
   ↓ 통과
2. S-MODEL-2 (실루엣 구분)     ← 실패 시 해당 모델 재생성
   ↓ 통과
3. S-MODEL-3 (스타일 통일)     ← 실패 시 해당 모델 재생성
   ↓ 통과
4. S-MODEL-4 (기능 스모크)     ← 실패 시 코드 디버깅
   ↓ 통과
5. S-MODEL-5 (성능)           ← 실패 시 폴리곤 축소
```

---

## 파이프라인 변경 사항 (Phase 2 대비)

- **Blender 후처리 제거**: 사용자가 Blender를 모르고 설치하지 않음
- **Hunyuan3D 직접 출력**: face_reducer + Low-Polygon Stylization Module로 폴리곤 조절
- **Godot 임포트 단순화**: GLB를 project/assets/models/에 복사 → 자동 임포트
- **폴백 강화**: Blender 없으므로 AI 재생성 (최대 5회)이 주요 품질 개선 방법
